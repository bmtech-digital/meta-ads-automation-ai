## 1. Executive Summary

### Problem Statement

Campaigner works — the agent paradigm is sound, the HITL queue is right, the two-gate evaluation philosophy is the product. The system suffers from **organizational debt, not architectural breakage**. Three of eight documented flows (F/G/H) are written but not deployed. Guardrail logic exists in both markdown prose and Python enforcement code with no generation pipeline between them. Numeric thresholds are buried inside prose paragraphs where they cannot be tuned cleanly or version-stamped on the proposals they shape. Section-number markdown headings act as a fragile cross-file API. The personality doc addresses an interactive Claude conversation that does not exist in production. Forward-looking commitments are stored as Hebrew prose and regex-parsed back into agent memory.

The cumulative effect: a contributor arriving with "I want to change how the agent thinks about X" has no map. They grep across `CAMPAIGNER.md`, `performance-brain.md`, `decision-tree.md`, and `guardrails.md`, hope they found every reference, and edit four files praying they did not drift. The brain is editable in principle and unmaintainable in practice.

### Proposed Solution

Reorganize the brain into four explicit layers — markdown for judgment, YAML for numbers and wiring, Python CLI tools as the I/O surface, Python contracts for deterministic enforcement — with a **source-of-truth map** declaring exactly one home per kind of fact and a **generated concept index** mapping every reasoning concept to the file that owns it. Split `CAMPAIGNER.md` into per-flow prompt files plus a shared-brain layer for cross-flow reasoning. Make stable string slugs (not section numbers) the cross-file API. Deploy the three undeployed flows as a side effect of building a flow registry. Make `make generate` the build pipeline that emits CronJob manifests, the guardrails reference doc, and the concept index from their sources of truth.

The migration ships as **nine independently-shippable steps**, sequenced so each one leaves the system in a more navigable state than before. Step 4 (per-flow split + shared brain) ships as a single atomic change — the two halves are not independently shippable. The agent paradigm, the HITL queue, the two-gate philosophy, and the Python CLI tool surface are **explicitly preserved** (§3 Non-Goals).

### Success Criteria

Three tiers. The migration is declared done when all three are met.

**Tier 1 — Structural (one home per fact):**

- Every numeric threshold currently in prose is referenced by name from markdown and defined exactly once in `config/thresholds.yaml`. A `grep` for hardcoded numerics inside `campaigner/prompts/**/*.md` returns zero hits in the lanes the migration covers (the inventory is captured during Step 3 and enforced thereafter).
- Every deployed flow has exactly one source of wiring: a block in `config/flows.yaml`. CronJob manifests under `kubefiles/` are generated from it; hand-edited manifests are rejected at review.
- Every cross-file reference between markdown lanes / guardrails / flow definitions uses a stable string slug. Section numbers may appear in headings for human navigation but no code, prompt, or runner script depends on them.
- `prompts/CONCEPTS.md` is generated by `make generate` from frontmatter slugs and lists, for every named concept, its owner file and consumer files. Hand-edits to this file are rejected.
- `prompts/guardrails.md` is generated from `check_guardrails.py` rule definitions. Drift between the two is a build failure.
- The interactive-Claude persona is removed from the root `CLAUDE.md` and `docs/PERSONALITY.md`. No `CLAUDE.md` at any level assigns Claude a role.

**Tier 2 — Operational (the system runs better):**

- Flows F (weekly self-audit), G (daily A/B-test decisions), and H (midday health check) run on cron in production for **14 consecutive days** without manual intervention, producing `agent_decisions` rows at the same hygiene level as the four currently-deployed flows.
- The structured `plans` table is populated by every flow that writes a `תוכנית:` line in Hebrew rationale. The Hebrew line is for the operator; the row is for the agent's future self. `load_active_plans.py` queries the table — it no longer parses prose.
- Every `agent_decisions` row carries the `thresholds.yaml` schema version that was loaded for that run. "Why did this proposal fire on date X?" is answerable by joining the row to the matching threshold snapshot.

**Tier 3 — Editorial (the brain is editable):**

- A contributor arriving with "I want to change how fatigue is detected" reads `prompts/CONCEPTS.md`, finds the owner file, and edits exactly that file. The test: the change does not require touching `CAMPAIGNER.md`, more than one flow file, or any code. If it does, the filing system is wrong; consolidate before changing.
- A contributor arriving with "I want to tune the winner ratio from 0.85 to 0.80" edits exactly one line in `config/thresholds.yaml`. The next agent run picks up the new value and stamps the new schema version on its decisions.
- A contributor arriving with "I want to add a new flow" writes one block in `config/flows.yaml`, one file under `prompts/flows/<name>.md`, runs `make generate`, and ships. No CronJob YAML is hand-written.
- The 12 editing rules in §5 are linked from the root `CLAUDE.md` navigation table. PR review references them by number.

**Cost is not a gate.** The migration's purpose is editorial discipline; if token cost drops as a side effect of the per-flow load matrix, that is a bonus, not a target.

## 2. The Mental Model

The migration's substance is not "nine steps." The substance is the **mental model** the nine steps install. The steps are the build order; the model is what survives after.

### 2.1 Four layers

Every piece of the brain belongs to exactly one of four layers. Knowing which layer a piece belongs to tells you which file to edit when it changes.

| Layer                              | Where it lives                                                       | What it holds                                                                            | Editor's discipline                                                                                |
| ---------------------------------- | -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------- |
| **1. The agent's brain**           | `campaigner/prompts/` + the residual `campaigner/CAMPAIGNER.md`      | Diagnostic methods, decision-tree lanes, per-flow protocols, voice rules                 | Markdown is for judgment. If the change is a number, it belongs in Layer 2.                        |
| **2. Numbers and configuration**   | `config/thresholds.yaml`, `config/flows.yaml`                        | Every numeric threshold; every flow's schedule / runner / prompt-load matrix             | YAML is the source of truth. Markdown references thresholds by name; literal copies in prose drift.|
| **3. CLI tools**                   | `campaigner/tools/`                                                  | Single-purpose, agent-callable, contract-bound. The agent's I/O surface.                 | No business logic. Each tool emits one JSON blob with a stable shape.                              |
| **4. Code-enforced contracts**     | `campaigner/tools/check_guardrails.py`, `execute_task.py`, `lib/`    | Deterministic enforcement: guardrails, schema validation, payload contracts              | Small on purpose. A check enters Layer 4 only when it is genuinely deterministic.                  |

### 2.2 Source-of-truth map

For every kind of fact, exactly one file is the source. Other copies are **generated**.

| Kind of fact                                      | Source of truth                                          | Generated copies                                                          |
| ------------------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------------------- |
| Schedule, runner, prompt-load matrix for a flow   | `config/flows.yaml`                                      | `kubefiles/agent_cronjob_*.yaml`; the flow table in `CAMPAIGNER.md`        |
| Numeric thresholds                                | `config/thresholds.yaml`                                 | Markdown references the threshold by name; no literal copies              |
| Deterministic guardrails                          | `campaigner/tools/check_guardrails.py`                   | `campaigner/prompts/guardrails.md` (auto-generated reference)             |
| Decision-tree lanes                               | `prompts/decision-tree.md`                               | Cross-referenced by stable slug; no parallel code definition              |
| Voice & personality rules                         | `prompts/hebrew-copy-style.md`                           | None                                                                      |
| Per-flow protocol                                 | `prompts/flows/<flow_name>.md`                           | The flow index in residual `CAMPAIGNER.md`                                |
| Cross-flow reasoning                              | `prompts/shared-brain.md`                                | Referenced from per-flow files by stable slug                             |
| Concept-to-file index                             | Frontmatter slugs in each `.md` definition               | `prompts/CONCEPTS.md` (auto-generated by `make generate`)                 |
| Database schema                                   | `migrations/*.sql`                                       | None                                                                      |
| Forward-looking commitments                       | `plans` Postgres table                                   | Quoted in Hebrew rationale as `תוכנית:` lines for human readers           |

### 2.3 Per-flow prompt template

Each flow lives in its own file under `campaigner/prompts/flows/<flow_name>.md`, following a uniform template: **Identity → When this runs → Inputs available (Tools / Data sources / Reference docs) → What to check → What to compare → Decision rules → Constraints → Plans this flow consumes or creates → Outputs required (DB / stdout / exit code) → Edge cases → Worked example**. Roughly 300-500 lines per file. Together with the shared brain (§2.4), they replace most of `CAMPAIGNER.md`, which shrinks to a thin index plus universal preamble (~100 lines).

### 2.4 The shared brain layer

Between `CAMPAIGNER.md` (thin index + universal preamble) and `prompts/flows/<flow_name>.md` (per-flow protocols) sits `prompts/shared-brain.md` — the home of any reasoning used by more than one flow. The two-gate model, fatigue detection logic, portfolio-rebalance heuristics, the diagnostic method itself, lane definitions used by multiple flows. The per-flow files describe **when and how** their flow applies the shared reasoning, and add only what is genuinely flow-specific.

The discipline that keeps this honest is Rule 11 (§5). Additions to the shared brain must demonstrate two-flow use; single-flow concepts stay in the flow file. Without that rule, the shared brain becomes the path of least resistance and devolves into the new `CAMPAIGNER.md` problem under a different name. The load matrix in `flows.yaml` declares whether each flow loads the shared brain (most do); per-flow files reference its concepts by stable slug.

### 2.5 Concept index

`prompts/CONCEPTS.md` is a generated artifact: for every named concept in the brain (slugged via frontmatter), it lists the owner file and the consumer files. A worker arriving with "I want to change how fatigue is detected" reads `CONCEPTS.md`, finds `fatigue_detection → shared-brain.md (owner); flows/daily_observe.md, flows/weekly_creative.md (consumers)`, and knows exactly what to edit and what to verify.

This is a different artifact from the per-folder `CLAUDE.md` files. Those are **navigation** indices: "here is what this folder does, here is its contract." `CONCEPTS.md` is the orthogonal axis: "here is where reasoning about X lives, regardless of folder." Both exist; neither replaces the other.

### 2.6 Plans as structured objects

A `plans` Postgres table. Each row carries a structured trigger (metric, operator, threshold-by-name, sustained-days) and a `proposed_action` payload template. When the agent commits to a forward-looking step, it writes a row **and** the Hebrew `תוכנית:` line. The row is for the agent's future self; the line is for the operator. Subsequent runs query the table — they do not parse prose. Operators get a `/plans` UI page.

### 2.7 Stable slugs

Cross-file references use stable string identifiers: `scale_up_candidate`, `creative_pool_exhausted`, `no_new_creative_when_underspending`, `daily_observe_propose`. Markdown headings can be renumbered freely; the slug is the API.

## 3. User Experience & Functionality

### 3.1 Personas

| Persona              | Role                                                                                        | How they touch the brain                                                                                                                                                                            |
| -------------------- | ------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Operator (Roi)**   | Product owner. Approves direction shifts; does not edit prompts.                            | Reads `agent_decisions` and `approvals`. Asks for tuning by saying "the agent should X more often"; the brain editor translates that into the right layer.                                          |
| **Brain editor**     | Contributor (human or Claude session) executing a migration step or a post-migration change.| Reads `prompts/CONCEPTS.md` first; edits the owner file; runs `make generate`; runs the relevant golden in `tests/golden/`. Never edits a generated artifact.                                       |
| **Reviewer**         | PR reviewer enforcing the editing rules in §5.                                              | Rejects PRs that hand-edit generated artifacts, that add a hardcoded number to prose, that touch more than one file for a single conceptual change, or that add to the shared brain without two-flow use.|
| **Future Claude**    | The agent at runtime, during a cron-driven flow.                                            | Loads the per-flow file declared in `flows.yaml`; loads `shared-brain.md` if the flow's matrix says to; loads `thresholds.yaml` at the start of every run and stamps its version on every decision. |

### 3.2 User stories (change scenarios)

The PRD's user stories are not "as an end user, I want to click X." They are **change scenarios** — the situations the migration is optimizing for. A scenario the post-migration system handles cleanly is a success; one it does not is a failure of the filing system.

**US-1 — Tune a number.**
_As a brain editor, I want to change the winner ratio from 0.85 to 0.80 by editing one line in `config/thresholds.yaml`, so I do not have to grep across markdown for the literal `0.85` and pray I found every copy._

**AC:**
- The literal `0.85` does not appear in any markdown file under `campaigner/prompts/` after Step 3 lands. Markdown references the threshold by its YAML name.
- A change to `thresholds.yaml` requires no other file change for the agent to pick it up on the next run.
- The new value is stamped on every `agent_decisions` row written after the change; the old value remains queryable for rows written before.

**US-2 — Add a flow.**
_As a brain editor, I want to add a new "biweekly competitor pricing scan" flow by writing one block in `config/flows.yaml` and one file under `prompts/flows/`, running `make generate`, and shipping. No CronJob YAML is hand-written._

**AC:**
- `config/flows.yaml` is the only place a flow's schedule, runner, and prompt-load matrix are declared.
- `make generate` emits the matching CronJob manifest under `kubefiles/` and the flow row in the `CAMPAIGNER.md` flow index.
- Hand-edited CronJob manifests are rejected at review (Rule 6).

**US-3 — Revise a piece of reasoning.**
_As a brain editor, I want to change how fatigue is detected by editing exactly one file. I read `prompts/CONCEPTS.md`, find `fatigue_detection → shared-brain.md (owner); flows/daily_observe.md, flows/weekly_creative.md (consumers)`, edit `shared-brain.md`, run the relevant goldens, and ship._

**AC:**
- `CONCEPTS.md` lists, for every slugged concept, owner + consumers with file:line references.
- A concept used by ≥2 flows lives in `shared-brain.md`. A concept used by 1 flow lives in that flow's file. Promotion happens on second use; demotion is equally legitimate (Rule 11).
- The change does not require touching `CAMPAIGNER.md`, multiple flow files, or any Python code. If it does, the filing system is wrong; the brain editor consolidates first, then changes.

**US-4 — Add a deterministic guardrail.**
_As a brain editor, I want to add a new guardrail rule by adding a function to `check_guardrails.py` plus a test. The markdown reference at `prompts/guardrails.md` regenerates automatically on `make generate`._

**AC:**
- The rule's logic lives only in `check_guardrails.py`. The markdown reference is generated; hand-edits to `prompts/guardrails.md` are rejected.
- A unit test for the new rule lives under `tests/lib/` or `tests/tools/` and runs in CI.
- The rule is keyed by a stable slug. Existing prompts may reference it by slug; section numbers in the generated markdown are decorative.

**US-5 — Commit to a forward-looking plan.**
_As the agent, when I write "תוכנית: if utilization climbs back above 80%, propose a scale-up" in a rationale, I also write a structured row to the `plans` table so my future self can query the commitment instead of re-parsing prose._

**AC:**
- `propose_task.py` accepts a `--plan` argument carrying the structured trigger (metric, operator, threshold-by-name, sustained-days) and the proposed-action payload template.
- `load_active_plans.py` queries the `plans` table; the legacy prose-parsing path is removed once Step 5 lands.
- Operators see active plans in a `/plans` UI page (delivered as part of the frontend; backend writes the table).

**US-6 — Onboard.**
_As a new contributor, I read the root `CLAUDE.md`, follow its navigation table, and land on `shared-brain.md`, `flows/<flow_name>.md`, `CONCEPTS.md`, and `thresholds.yaml` — not on the residual sections of `CAMPAIGNER.md` that no longer carry weight._

**AC:**
- The root `CLAUDE.md` navigation table points at the new structure (Step 9).
- The per-folder `CLAUDE.md` files under `campaigner/`, `campaigner/prompts/`, `campaigner/tools/`, and `campaigner/cli/` have "Where truth lives" sections that match the source-of-truth map in §2.2.
- Older prose pointing at the legacy spec is updated in the same PR that retires the legacy spec section.

### 3.3 Non-Goals (preserve the agent paradigm)

To protect the agent paradigm from drift during the migration, these are **explicitly not changing**:

- **Claude reads markdown, reasons, calls tools, writes output.** The LLM stays at the top of the stack. No flow becomes a deterministic pipeline.
- **The HITL approval queue.** Every action still writes to `approvals` and waits for human approval before any Meta write. The execute flow still re-checks guardrails at execute time.
- **The two-gate evaluation philosophy.** Leading signals at the creative level in 48h-7d windows; lagging signals at the campaign level post-Learning. The philosophy moves into `shared-brain.md`; it is not revised.
- **The Python CLI tools as the agent's I/O surface.** Each tool stays single-purpose, contract-bound, and emits one JSON blob. The catalog can grow; the contract does not.
- **The audit log (`agent_decisions`) and the heartbeat-based observability.** Schema may grow (e.g., threshold-version stamping) but never shrinks.
- **The Hebrew-first voice for operator-facing output.** Voice consolidates into one file (`prompts/hebrew-copy-style.md`); the language and tone are not changed.
- **LangGraph orchestration.** Deferred to v2. v2 is triggered by adding a second ad account, not by this migration. Reorganizing the brain does not migrate to LangGraph.

## 4. Migration Requirements (the nine steps)

Each step is independently shippable and leaves the system in a more navigable state than before. **Step 4 is the exception** — its two halves (per-flow split + shared-brain extraction) ship together. Effort scale: small ≈ 1-2 days, medium ≈ 3-7 days, large ≈ 2-3 weeks.

### Step 1 — Strip the interactive persona

| Attribute      | Value                                                                                                                                                                       |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**     | Small                                                                                                                                                                       |
| **Risk**       | Low                                                                                                                                                                         |
| **Unlocks**    | Clean separation between production agent and interactive Claude. Closes Finding 5.                                                                                         |
| **Dependencies** | None                                                                                                                                                                      |

**Scope.** Delete the `🎙️ How You Talk — Personality (binding)` section from the root `CLAUDE.md`. Rewrite `docs/PERSONALITY.md` to voice-only content, or fold its surviving rules into `prompts/hebrew-copy-style.md`. Update navigation tables in the various per-folder `CLAUDE.md` files to remove pointers to the deleted persona.

**Acceptance Criteria.**
- No `CLAUDE.md` at any level assigns Claude a role beyond "coding assistant for this codebase."
- `docs/PERSONALITY.md` either no longer exists or contains only voice rules that also appear in `prompts/hebrew-copy-style.md` (single source of truth — Rule 5).
- Interactive sessions in this repo no longer adopt a campaign-diagnostician persona.

### Step 2 — Build the flow registry

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low-medium                                                                                                                                                                  |
| **Unlocks**      | Flows F (weekly self-audit), G (daily A/B-test decisions), H (midday health check) deployed; no more wiring drift. Closes Finding 1.                                       |
| **Dependencies** | None                                                                                                                                                                        |

**Scope.** Create `config/flows.yaml` with all eight flows: name, schedule, timezone, runner, trigger prompt, prompts-loaded matrix, description. Build a `make generate` target that produces (a) CronJob manifests under `kubefiles/`, (b) the flow table in `CAMPAIGNER.md`, and (c) the flow index in `docs/ARCHITECTURE.md`. Deploy F/G/H as a side effect of regenerating from the new source. Runner scripts read their prompt-load matrix from `flows.yaml` rather than hardcoding paths.

**Acceptance Criteria.**
- `config/flows.yaml` contains a block for every deployed flow. Adding a flow means adding a block plus a per-flow prompt file (Step 4).
- `make generate` is idempotent and exits clean on no-op.
- Hand-edited CronJob manifests are rejected at review (Rule 6). A CI check diffs the on-disk manifest against the generated output.
- Flows F, G, H run on cron in production for 14 consecutive days post-deploy with the same hygiene as the four currently-deployed flows.

### Step 3 — Extract thresholds

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | Tunable without code change; version-stamped retroactive answers to "why did this proposal fire?". Closes Finding 3.                                                       |
| **Dependencies** | None (parallel-safe with Step 1, 2, 5)                                                                                                                                      |

**Scope.** Create `config/thresholds.yaml`. Walk the prompts and pull every hardcoded number into the YAML, replacing the literal with a named reference. Add a `load_thresholds.py` tool and update `CAMPAIGNER.md` Step 1 to load thresholds at the start of every run. Stamp the schema version on every `agent_decisions` row.

**Inventory (non-exhaustive, captured during the step):**
- Anti-flood caps (`< 50 → 2, 50-500 → 5, > 500 → 10`) — currently in `CAMPAIGNER.md` Step 5.
- Winner ratio (`CPA ≤ target × 0.85`) — currently in `performance-brain.md` §5.
- Utilization floor for new-creative suppression (`utilization_7d < 0.5`) — currently in `guardrails.md` §19.
- The "average lane" baseline band (KPI ±15%) — currently in `performance-brain.md` §5.

**Acceptance Criteria.**
- A grep for hardcoded numeric literals inside `campaigner/prompts/**/*.md` returns zero hits in the inventory above (and any others surfaced during the walk).
- `thresholds.yaml` has a schema version. Every `agent_decisions` row written post-Step-3 carries the version that was loaded for that run.
- Markdown references the threshold by its YAML name; the literal value never appears in prose.
- Changing a threshold requires editing exactly one line of YAML; the next agent run picks it up.

### Step 4 — Per-flow prompt files + shared brain

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Large                                                                                                                                                                       |
| **Risk**         | Medium                                                                                                                                                                      |
| **Unlocks**      | Navigable per-flow docs; a single home for cross-flow reasoning; token-cost relief. Closes Findings 8, 10.                                                                  |
| **Dependencies** | Step 2 (the load matrix in `flows.yaml` is where each flow declares which files it loads)                                                                                  |

**Scope.** Split `CAMPAIGNER.md` into `prompts/flows/<flow_name>.md` files following the template in §2.3. **In the same pass**, extract every concept used by more than one flow into `prompts/shared-brain.md` — the two-gate model, fatigue detection, portfolio rebalance, the diagnostic method, lane definitions referenced from multiple flows. Per-flow files describe when and how their flow applies the shared reasoning and add only what is genuinely flow-specific. Shrink `CAMPAIGNER.md` to its residual index plus universal preamble (~100 lines). Update runner scripts' trigger prompts (via `flows.yaml`) to reference the new files.

**Atomicity note.** Splitting per-flow **without** the shared-brain extraction is not independently shippable — it distributes shared reasoning across N files without a home, creating exactly the drift risk Finding 10 warns about. The two halves of Step 4 ship together. Run goldens against the post-split brain before merging.

**Acceptance Criteria.**
- Every deployed flow has a corresponding `prompts/flows/<flow_name>.md` following the template (Identity → When this runs → Inputs → … → Worked example).
- `prompts/shared-brain.md` exists. Every concept used by ≥2 flow files lives in it; per-flow files reference shared concepts by stable slug (cf. Step 6).
- `CAMPAIGNER.md` shrinks to ~100 lines: thin index + universal preamble. Sections that moved have either been deleted or replaced with a single line pointing at the new home.
- All currently-passing goldens under `tests/golden/` still pass on the post-Step-4 brain.

### Step 5 — Structured plans table

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | Queryable agent memory; `/plans` UI page. Closes Finding 6.                                                                                                                |
| **Dependencies** | None (parallel-safe with Steps 1, 2, 3)                                                                                                                                     |

**Scope.** Add the `plans` migration: trigger (metric, operator, threshold-by-name, sustained-days), `proposed_action` payload template, owning-flow, status (`active`, `fired`, `expired`, `withdrawn`). Update `propose_task.py` to accept `--plan` and write the row alongside the approval. Update `load_active_plans.py` to query the table. Optionally backfill from existing Hebrew rationale; otherwise let open prose-plans expire naturally.

**Acceptance Criteria.**
- Every flow that writes a `תוכנית:` line in Hebrew rationale also writes a structured `plans` row.
- `load_active_plans.py` queries the table; the prose-parsing path is removed.
- The trigger references threshold values by name (Step 3), not by hardcoded literal.

### Step 6 — Stable slugs

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | Safe renumbering; clearer cross-references. Closes Finding 4. **Required by Step 8.**                                                                                       |
| **Dependencies** | Step 4 (the per-flow structure is what gets slugged)                                                                                                                       |

**Scope.** Add slugs as frontmatter to existing lane and guardrail definitions. Walk every cross-reference (in code, in prompts, in tests, in runner trigger strings) and migrate from section numbers to slugs. Section numbers may remain in markdown headings for human navigation but stop being load-bearing.

**Acceptance Criteria.**
- Every lane / guardrail / shared-brain concept that is referenced from elsewhere carries a frontmatter slug.
- No code path, prompt, or runner string references a markdown section number for behavior.
- A renaming exercise (rename a heading) requires no other file change. CI verifies via the duplicate-slug / dangling-slug check (Step 8).

### Step 7 — Generated reference docs

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | No drift between code and prose. Closes Finding 2.                                                                                                                          |
| **Dependencies** | Step 6 (rules are keyed by slug, not section number)                                                                                                                       |

**Scope.** Auto-generate `prompts/guardrails.md` from `check_guardrails.py`'s rule definitions (docstrings, slugs, parameters). Auto-generate the lane reference from `decision-tree.md`'s frontmatter. Wire both into `make generate`. The hand-edited markdown reference goes away.

**Acceptance Criteria.**
- `prompts/guardrails.md` is regenerated by `make generate` and matches `check_guardrails.py` byte-for-byte in CI.
- Hand-edits to `prompts/guardrails.md` are rejected at review (Rule 1).
- The lane reference in the generated artifact stays in sync with `decision-tree.md`'s frontmatter on every run.

### Step 8 — Generated concept index

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Medium                                                                                                                                                                      |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | "I want to change X" becomes one file-lookup, not a grep across the brain. Closes Finding 10.                                                                              |
| **Dependencies** | Step 6 (slugs across the brain) and Step 4 (the per-flow / shared-brain structure that gets indexed)                                                                       |

**Scope.** Extend `make generate` to walk the markdown tree under `prompts/` and emit `prompts/CONCEPTS.md`: for every slug, list owner file + line, consumer files + lines. Fail the build on:
- a duplicate slug,
- a slug referenced but never defined,
- a definition with no slug,
- (optional, post-launch) a concept in `shared-brain.md` used by fewer than two flows.

**Acceptance Criteria.**
- `prompts/CONCEPTS.md` is a generated artifact. Hand-edits are rejected.
- The build fails on slug violations listed above.
- A contributor changing a concept reads `CONCEPTS.md` first and edits exactly the owner file. US-3 passes end-to-end.

### Step 9 — Onboarding cut-over

| Attribute        | Value                                                                                                                                                                       |
| ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Effort**       | Small                                                                                                                                                                       |
| **Risk**         | Low                                                                                                                                                                         |
| **Unlocks**      | The new structure is the entry point a worker actually finds.                                                                                                              |
| **Dependencies** | Steps 1, 4, 6, 8 (the structure being navigated must exist before the navigation table points at it)                                                                       |

**Scope.** Rewrite the navigation table in the root `CLAUDE.md` and the "Where truth lives" tables in the per-folder `CLAUDE.md` files to point at `shared-brain.md`, `flows/<flow_name>.md`, `CONCEPTS.md`, and `thresholds.yaml`. Remove pointers to the residual sections of `CAMPAIGNER.md` that no longer carry weight. A new contributor reading the root `CLAUDE.md` should land on the shared brain or the concept index, not on the legacy spec.

**Acceptance Criteria.**
- A new contributor following the root `CLAUDE.md` navigation reaches a per-flow file or `CONCEPTS.md` within two clicks.
- The 12 editing rules (§5) are linked from the navigation table.
- Older prose pointing at the legacy spec is updated in the same PR that retires it.

## 5. Editing Discipline — the 12 Rules

These rules describe the post-migration target discipline. They can be adopted now (mid-migration); they do not depend on the migration being complete. PR review enforces them by number.

| #  | Rule                                                                                       | What it prevents                                                                                                                              |
| -- | ------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------- |
| 1  | **One change, one home.**                                                                  | A change that requires touching two files signals the filing system is wrong — consolidate first, then change.                                |
| 2  | **Numbers belong in YAML.**                                                                | Hardcoded numerics inside markdown paragraphs drift silently and cannot be version-stamped on the proposals they shape.                       |
| 3  | **Markdown is for judgment; code is for what must be enforced.**                           | Backing every markdown rule with a code check inflates the deterministic layer and removes the agent's judgment from situations that need it. |
| 4  | **Stable slugs, not section numbers.**                                                     | Renaming a heading silently breaks references; section numbers look stable but are not.                                                       |
| 5  | **One voice file.**                                                                        | Voice rules split across files diverge; the production agent and the interactive Claude end up with different personas.                       |
| 6  | **The flow registry is the wiring authority.**                                             | Hand-edited CronJob manifests drift from the load matrix and from `CAMPAIGNER.md`'s flow index.                                              |
| 7  | **Additions replace; they do not accrete.**                                                | Amended-paragraph layers cannot be reasoned about. Rewrite, do not append.                                                                    |
| 8  | **Token weight is editorial discipline.**                                                  | An always-loaded prompt is paid for on every tool turn. A 500-line addition to `decision-tree.md` is paid 76× per Flow A run.                |
| 9  | **Plans are structured, not parsed.**                                                      | A regex over Hebrew prose is the agent's least-reliable memory layer.                                                                         |
| 10 | **Test fixtures for deterministic layers, golden runs for the agent layer.**               | Unit tests for `check_guardrails.py`; goldens for the reasoning layer. The two test layers exist for different reasons.                       |
| 11 | **The shared brain is for cross-flow reasoning only.**                                     | Without this, the shared brain becomes the path of least resistance and devolves into the new `CAMPAIGNER.md` problem under a different name. |
| 12 | **The concept index is generated, not written.**                                           | A hand-edited `CONCEPTS.md` is a lie. The fix is to add a slug to the definition, not to edit the index.                                       |

## 6. Risks & Roadmap

### 6.1 Sequencing

The nine steps are independently shippable except where dependencies are noted below. The recommended sequence threads dependencies and front-loads low-risk wins:

```
Step 1 (persona)     ─┐
Step 2 (flow reg)    ─┼─► Step 4 (split + shared brain) ──► Step 6 (slugs) ──┬─► Step 7 (gen guardrails)
Step 3 (thresholds)  ─┤                                                       └─► Step 8 (concept index) ──► Step 9 (onboarding)
Step 5 (plans table) ─┘
```

Steps 1, 2, 3, 5 are parallel-safe and can ship in any order. Step 4 needs Step 2 (the load matrix declares what each flow loads). Step 6 needs Step 4 (the slug-bearing structure must exist). Steps 7 and 8 both need Step 6. Step 9 needs the structure it points at — practically, it ships last.

### 6.2 Technical risks

| Risk                                                                                                  | Likelihood | Mitigation                                                                                                                                                                                  |
| ----------------------------------------------------------------------------------------------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Shared brain devolves into the new `CAMPAIGNER.md`** — every new concept drifts upward into it.    | Medium     | Rule 11 + Step 8's optional check (a concept in `shared-brain.md` used by fewer than two flows fails the build). Start in the flow file; promote on second use; demote when only one remains. |
| **Golden runs miss a regression introduced by the Step 4 split.**                                     | Medium     | Run goldens against the post-split brain before merging Step 4. Add a "shadow" run of Flow A against the old and new brain side-by-side for the first week post-merge.                       |
| **Hand-edits to generated artifacts slip in via PR.**                                                 | Medium     | CI check: regenerate, diff, fail on drift. Rule 1 and Rule 12 in PR template's reviewer checklist.                                                                                          |
| **Threshold version stamping breaks retroactive comparisons** when the YAML schema changes shape.    | Low        | Treat `thresholds.yaml` schema versions like migrations — additive when possible, version bump otherwise. Store the full snapshot (not just the version number) once per version.            |
| **Plans table backfill is incomplete** — open prose-plans are lost when `load_active_plans.py` cuts over. | Low-medium | Step 5 backfill is optional; the cut-over is staged. Run the prose-parsing path in shadow mode for one week post-Step-5 deploy and reconcile.                                            |
| **Flow F/G/H deploy reveals prompt bugs** that were hidden by non-deployment.                         | Medium     | Deploy F/G/H one at a time after Step 2 lands. Watch `agent_decisions` for the first 3 days. Treat the deploy as Phase-2 backend work, not Step-2 work.                                    |

### 6.3 Organizational risks

- **The migration takes longer than its individual steps suggest** because the editorial discipline (Rules 7, 11) is a behavior change, not a code change. Mitigation: review the rules at the start of each PR, not at the end.
- **The interactive persona is missed by someone who liked it.** Mitigation: the operator decided to remove it (Finding 5; the production agent is headless and cannot use it). Step 1's PR should link the audit finding.
- **Step 4 is the only large-effort step.** A two-week stall risk. Mitigation: scope it as a single PR but allow the per-flow file extraction and shared-brain extraction to land as two commits on the same branch, reviewed together.

### 6.4 Phased rollout

- **Phase A — Independent foundations (Steps 1, 2, 3, 5):** Ship in any order. Each lands a self-contained improvement and prepares the substrate for Step 4.
- **Phase B — Brain split (Step 4):** Single atomic change. Goldens guard the regression surface.
- **Phase C — API + generation (Steps 6, 7, 8):** Slugs first, then both generators. Step 8 closes the "I want to change X" gap and is the migration's headline outcome.
- **Phase D — Onboarding cut-over (Step 9):** The new structure becomes the entry point. The legacy spec sections are retired in the same PR.

## 7. Out of scope (deferred or rejected)

- **LangGraph migration.** v2. Triggered by a second ad account, not by this migration.
- **A second observability stack** (LangSmith / Langfuse). `agent_decisions` remains the observability surface. Threshold-version stamping (Step 3) is the only schema change required by this migration.
- **A web UI for editing prompts.** Editing happens in the repo. The `/plans` UI page (Step 5) reads the plans table; it does not write reasoning.
- **A formal RFC process for brain changes.** The 12 editing rules are the discipline. Rule 7 ("additions replace, they do not accrete") substitutes for an RFC layer.
- **Renaming projects, flows, or tools for stylistic consistency.** The migration touches structure, not names.
